# AutoLib — 南京林业大学图书馆自动抢座系统

> 原作：[lnk137/NJFU-AutomaticLibrary](https://github.com/lnk137/NJFU-AutomaticLibrary)，基于此二创的 Web 版本。

---

## 这个项目是干什么的？

每天抢图书馆座位太麻烦？AutoLib 帮你**全自动搞定**。

你只需要在网页上配置一次：学号、密码、想坐哪、几点到几点。之后每天到了预约时间，系统会自动登录你的账号，帮你抢好座位，抢完还能发通知告诉你结果。

---

## 主要功能

### 自动抢座

- 每天在你设定的时间（默认 19:55）自动执行预约
- 可以配置多个备选座位，抢到一个就停，抢不到就依次往下试
- 支持按星期设置不同时间段（比如周一全天，周五只到 20 点）
- 支持一天预约多个时间段（早上一段 + 下午一段）

### 迟到保护

- 如果你今天起晚了，座位要被系统自动释放怎么办？
- 开启这个功能后，系统会在你的预约开始前 7 分钟检查你有没有签到
- 如果还没签到，自动帮你取消原预约、往后顺延重新预约，避免被记违约
- 如果你已经到馆签到了，就不会触发

### 午休重约

- 中午想午休，但不想放弃下午的座位？
- 点一下「午休」，系统自动取消上午的预约，立刻重新预约下午时段
- 也支持每天定时自动执行（比如每天 12:05 自动午休）

### 多账号管理

- 一个网站账号下可以绑定多个学号
- 每个学号独立配置预约规则、座位列表、通知方式
- 支持游客模式：不注册也能先试用，注册后数据自动合并

### 结果通知

- 预约成功或失败后，可以通过以下方式收到通知：
  - **邮件**：发到你填写的邮箱
  - **微信**：通过 Server 酱推送到微信

### 手动操作

如果你临时需要，也可以在网页上手动操作：
- 立即执行一次抢座（不等定时任务）
- 按自定义时间段预约今天
- 查询当前预约状态
- 取消某条预约

### 管理后台

- 管理员可以通过独立后台管理所有用户
- 发布公告（支持置顶、高亮级别）
- 查看、开关、删除用户的预约配置
- 调整迟到保护策略（保护时长、黑名单）
- 后台登录使用二步验证（TOTP），安全有保障

## 底层是怎么实现的？

### 整体架构

项目通过 **Docker Compose** 启动 4 个容器，互相协作：

```
┌─────────────────────────────────────────────────────┐
│                   Docker Compose                    │
│                                                     │
│  flask-api   ←→   MongoDB   ←→   scheduler          │
│  （网页+接口）     （数据库）      （定时任务）          │
│                     ↑                               │
│                   seed                              │
│              （一次性导入座位数据）                    │
└─────────────────────────────────────────────────────┘
```

| 容器 | 技术 | 作用 |
|------|------|------|
| `flask-api` | Python + Flask | 提供网页界面和 REST API |
| `mongo` | MongoDB 7 | 存储用户配置、预约结果、座位信息、公告 |
| `scheduler` | APScheduler | 每天定时执行预约任务、迟到保护、午休检查 |
| `seed` | Python | 启动时一次性将座位信息导入数据库 |

### 如何自动登录图书馆系统？

图书馆预约系统在校园网内网，校外（或关了 VPN）无法访问，所以需要先过 VPN 这一关。整个登录流程分两步：

**第一步：VPN 登录（统一身份认证）**

1. 访问学校 webVPN 登录页，获取表单中的加密盐值（`salt`）
2. 用 AES 算法加密密码，提交到 CAS 认证服务器
3. 认证成功后，HTTP Session 里会带上 VPN Cookie，后续请求就能穿透内网

**第二步：图书馆系统登录（IC 空间）**

1. 带着已登录的 VPN Session，访问图书馆系统，获取 RSA 公钥和 nonce（防重放随机串）
2. 用 RSA 公钥加密「密码 + nonce」
3. 发送登录请求，成功后设置 `ic-cookie`（含 token）
4. 之后的预约、查询、取消操作都带着这个 Cookie 发请求

```
用户密码
   ↓
[AES 加密] → VPN 登录 → VPN Cookie
                              ↓
                    [RSA 加密] → 图书馆登录 → ic-cookie
                                                  ↓
                                          预约 / 查询 / 取消
```

### 定时任务怎么跑？

`scheduler` 容器里运行一个独立的 Python 进程，用 **APScheduler** 管理调度：

- **每天 19:55（可配置）**：触发主预约流程
  1. 从 MongoDB 读取所有开启预约的用户配置
  2. 按优先级排序，逐一执行预约
  3. 预约结果写回数据库，同时发通知
  4. 预约完成后，启动迟到保护服务（会一直运行到当晚 22:00）

- **每分钟扫描一次**：检查哪些用户到了午休触发时间，依次执行自动午休

迟到保护的工作方式：
- 读取数据库里每个用户今天的预约信息（座位名、预约时间、uuid）
- 在预约开始前 7 分钟注册一个一次性任务
- 到点时检查用户是否已签到（签到状态码 1093/3141），没签到就执行迟到保护操作

### 密码怎么保存的？安全吗？

用户的 VPN 密码和图书馆密码需要存下来（因为自动登录时要用），所以用的是**可逆加密**（AES），而不是哈希。

- 加密密钥来自环境变量 `ENCRYPTION_KEY`，不放在代码里
- 所有密码字段在存入数据库时加密，读出来后才解密，管理员也看不到明文
- 网站登录密码（Web 账号）用的是不可逆的 bcrypt 哈希
- 管理后台使用 TOTP 二步验证（就是手机上的动态验证码）

### 技术栈汇总

| 层次 | 技术 |
|------|------|
| 后端框架 | Python 3 + Flask |
| 数据库 | MongoDB（通过 PyMongo 操作） |
| 前端 | 原生 HTML / CSS / JavaScript |
| 定时调度 | APScheduler |
| 密码安全 | AES 可逆加密（用户凭据）+ bcrypt（Web 密码）|
| 管理员认证 | TOTP 动态验证码（pyotp）|
| 通知推送 | SMTP 邮件 + Server 酱（微信）|
| 部署 | Docker + Docker Compose |

## 快速开始

详细部署步骤见 [quickstart.md](quickstart.md)。

简要流程：

```bash
# 1. 复制并填写环境变量
cp .env.example .env
# 编辑 .env，填写 MONGO_PASS、SECRET_KEY、ENCRYPTION_KEY、ADMIN_TOTP_SECRET

# 2. 启动所有服务
docker compose up -d --build

# 3. 打开浏览器
# 用户端：http://localhost:5004/
# 管理后台：http://localhost:5004/admin
```

## 注意事项

1. **VPN 密码 = 统一身份认证密码**（登 webVPN 用的那个），不是 Wi-Fi 密码
2. **图书馆密码 = IC 空间系统密码**，默认是 `njfu学号!`
3. 密码是可逆加密保存的，务必保管好 `.env` 文件，不要提交到公开仓库
4. 生产环境建议配合 Nginx/Caddy 加 HTTPS，并设置 `SESSION_COOKIE_SECURE=true`
5. 周五图书馆 20:00 关门，预约结束时间超过 20:00 的会自动截断
